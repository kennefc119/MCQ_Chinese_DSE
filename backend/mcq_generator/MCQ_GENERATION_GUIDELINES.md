# DSE 中文科 MC 題目生成指引（LLM 專用）

> **目的**：本文件為未來 LLM 在生成 DSE 中文科指定篇章 MC 題目時的完整操作手冊。  
> 涵蓋品質準則、防重複策略、出題技巧、選項設計、資料庫寫入規範等。  
> 最後更新：2026-06-04（基於 573 題全庫審計及 2015–2025 DSE 真題分析）

---

## 一、系統架構概覽

本系統有兩種出題模式：

1. **自動管線**（LangGraph 三角色）：Strategist → Drafter → Critic（最多 3 輪修訂），由 `mcq-gen run` 觸發
2. **手動批量**（單個 LLM 一次性扮演三角色）：直接撰寫完整題目並透過 `batch_questions.py` 寫入資料庫

本指引主要服務**模式 2（手動批量）**，但所有品質準則同樣適用於模式 1。

### 資料庫結構
- `dsemcq_questions`：題目主表（id, passage_id, stem, difficulty 1-5, source, is_active, critique_score）
- `dsemcq_question_options`：選項表（id, question_id, text, is_correct, explanation）
- `dsemcq_question_tags`：技能標籤（question_id, tag_id）
- `dsemcq_passages`：12 篇指定篇章（p01–p12）

## 二、題幹（Stem）書寫規範

### 必須遵守的格式規則

1. **必須包含篇章名稱**  
   題幹中必須明確提及所考篇章的名稱（用書名號 `《》` 括起），讓學生清楚知道考核哪篇文章。  
   - ✅ `根據《出師表》，下列哪一項最能反映文中關於賞罰處理的主張？`  
   - ✅ `李白〈月下獨酌〉中「舉杯邀明月，對影成三人」運用了哪種修辭手法：`（唐詩宋詞用子篇名）
   - ✅ `「不仁者，不可以久處約」中「約」的意思是：`（引用原文「」自帶辨識度亦可）
   - ❌ `下列哪一項最能概括本文的主旨？`（「本文」不行，學生不知道指哪篇）
   - ❌ `下列哪一項最能概括第三段的要旨？`（沒有篇章名，僅靠段號不足）

2. **禁止出現篇章內部編號**  
   永遠不要在題幹或選項中使用 `p01`、`p10`、`（p12）` 等內部 ID。這些是系統內部標識，學生看不到。  
   - ❌ `比較 p11（唐詩三首）與 p12（宋詞三首）`  
   - ✅ `比較《唐詩三首》與《宋詞三首》`

3. **語言莊重、學術化**  
   模仿 DSE 考評局的措辭風格：
   - 使用：「最能體現」、「何者最為貼切」、「試比較兩者……」、「下列哪一項」
   - 避免：口語化、過度文學化、命令式語氣

4. **題幹精煉、指向明確**  
   - 一題只考一種能力，不要混合
   - 避免雙重否定（「以下哪一個說法不是對X不正確的描述？」）
   - 如需用「不」字，只用一個否定且加粗

5. **引用原文用法**  
   引用篇章原文時，使用中文引號 `「」`，引文前後提供出處線索：
   - ✅ `孟子以「魚」與「熊掌」的取捨為喻，此比喻論證最主要的作用是：`
   - ✅ `廉頗揚言「我見相如，必辱之」，此言最能反映他當時怎樣的心態：`

## 三、選項設計準則

### 3.1 結構要求

- **4 個選項**：1 正確 + 3 干擾項
- **長度對稱**：四個選項字數誤差 ≤ 30%，句式結構一致
- **字數控制**：每選項 8–15 個中文字（除非考核完整句意義）
- **提取公因式**：如果四個選項都有相同的開頭/結尾字詞，移至題幹中
- **禁止**：「以上皆是 / 以上皆非 / 兩個選項皆可」

### 3.2 干擾項的高階設計（5 種 DSE 常見陷阱）

這是區分題目品質的關鍵。每個干擾項必須「看似合理但確切錯誤」，運用以下策略：

| 陷阱類型 | 說明 | 範例 |
|---|---|---|
| **偷換概念** | 挪用原文字眼但套用在錯誤對象或情境 | 「約」解為「約束自己」（原文指窮困） |
| **過度引申** | 基於原文事實推出文本未支持的極端結論 | 「主張人應捨棄一切物質追求」（原文只談生義取捨） |
| **部分正確** | 前半正確但後半因果關係或結論錯誤 | 「地位低下而遭人輕視」（前半對，後半引申了） |
| **以偏概全** | 將局部觀點說成全篇主旨 | 用次要論點冒充核心觀點 |
| **張冠李戴** | 將正確知識套在錯誤的對象上 | 「羽扇綸巾」指諸葛亮（實際在本詞中指周瑜） |

### 3.3 正確答案的要求

- 無懈可擊，完全符合原文及教師版工作紙的詮釋
- 用語必須高度吻合工作紙中的字眼（如「克制私慾」、「坦誠寬廣」）
- 字詞解釋題 100% 採用工作紙定義

## 四、解釋（Explanation）書寫規範

每個選項各自有一個 `explanation` 欄位：

### 正確選項的解釋
- 開頭：`此選項正確。`
- 必須引用篇章原文具體字句（用 `「」` 括起）
- 結合工作紙分析，清晰論證其正確性
- 範例：`此選項正確。原文「假輿馬者」意謂借助車馬的人，「假」通「借」，指憑藉外物之力，與下文「善假於物」的「假」同義。`

### 錯誤選項的解釋
- 開頭：`此選項錯誤。`
- 一針見血指出具體謬誤類型（偷換概念、過度引申等）
- 說明為何看似合理但實際不符
- 範例：`此選項錯誤。「簡約」為現代用法，原文「約」非指生活風格，而是指困窮的境遇。`

### 禁止事項
- ❌ 不使用 A、B、C、D 字母指代選項（系統會洗牌）
- ❌ 不在一個選項的解釋中提及其他選項
- ❌ 不寫空泛的「不符文意」，必須具體說明錯在哪裡

## 五、防重複策略（至關重要）

### 5.1 重複的定義

兩個題目被視為重複，當且僅當以下**兩個條件同時成立**：
1. 引用**完全相同或高度近似的同一句原文**
2. 測試**幾乎一模一樣的具體問題/角度**

以下**不算**重複：
- 引用不同句子，即使考核同一概念
- 引用同一句子，但考核不同角度
- 同一技能類別但引用不同句子

### 5.2 生成前的必做檢查

在撰寫任何題目之前，**必須先查詢現有題庫**：

```python
# 使用 mcq_gen/db/stems.py 的 fetch_existing_stems() 函數
# 或直接查詢 Supabase
from mcq_gen.db.client import get_supabase
sb = get_supabase()
result = sb.table("dsemcq_questions") \
    .select("id,passage_id,stem,difficulty") \
    .eq("is_active", True) \
    .execute()
```

按 passage_id 分組查看已有題幹，確保新題目：
- 不重複已有句子 + 已有考核角度的組合
- 優先填補已有題庫的空缺（passage × skill × difficulty 組合題數最少的）

### 5.3 從 2024 年審計中學到的教訓

**審計結果**：573 題中發現 98 組重複（共 243 題需要刪除），主要問題：
- p10（六國論）重複最嚴重：165 題中大量重複「賂秦根本原因」、「今日割五城」等
- 同一句子被反覆考核同一角度，僅措辭略有不同
- 缺乏多元化：某些 passage 的某些 skill 題數為 0

**防止重複的實操技巧**：
1. 寫題前先列出該篇章所有已有題幹
2. 每條新題目必須對應篇章中的**不同句子或不同段落**
3. 同一句子若已被考過某技能，改用另一技能切入
4. 善用 DSE 真題的提問角度，但轉換到不同的原文句子

## 六、DSE 真題出題模式與模仿技巧

### 6.1 DSE 甲部（指定篇章）常見題型

根據 2015–2025 年真題分析：

1. **字詞解釋**：直接問粗體字的意思，或問兩句中同一字的用法是否相同  
   → 我們的 MC 版：提供 4 個義項讓學生選擇

2. **跨篇章虛詞辨析**：「以下哪兩句的『以/之/而』用法相同？」  
   → 這是 DSE 最愛的 MC 題型之一

3. **篇章理解 + 分析**：「試就 XX 一文回答以下問題」  
   → 我們轉化為：「根據《XX》，下列哪一項最能……」

4. **修辭 + 作用**：不只問手法名稱，更問其在文中的效果  
   → 每個修辭題都要求辨識手法 + 說明作用

5. **人物性格推論**：從言行推論性格，而非直接查找描述  
   → 「此言最能反映他當時怎樣的心態」

### 6.2 模仿 DSE 的語言風格

| DSE 常用措辭 | 使用場景 |
|---|---|
| 「最能體現」 | 人物分析、主旨歸納 |
| 「何者最為貼切」 | 內容理解 |
| 「此語在文中最主要的作用是」 | 修辭手法、段落功能 |
| 「下列哪一項最能準確比較」 | 跨篇章比較 |
| 「試比較兩者在……的異同」 | 跨篇章比較 |
| 「根據……」 | 所有題型的開頭 |

### 6.3 避開 DSE 已考過的概念

系統已內建 `dse_reference.py`，會自動載入 2015–2025 年的真題作為參考。  
手動出題時，應先翻閱 `source/` 目錄下的歷屆試題 JSON，確保不重複真題角度。

## 七、難度設計技巧

### 最淺（difficulty=1）
- 直接字詞解釋（工作紙有列出的）
- 干擾項明顯但仍需辨析（不能太荒誕）
- 適合 p11/p12 的基本意象識別

### 淺（difficulty=2）
- 字詞在語境中的義項選擇
- 簡單的內容理解（單句或單段）
- 基礎修辭手法辨識（擬人、比喻等）

### 中（difficulty=3）
- 段落主旨歸納
- 人物行為動機分析
- 修辭手法 + 其在文中的具體作用
- 需要理解因果關係

### 深（difficulty=4）
- 跨段落整合
- 人物前後性格對比
- 「部分正確」型高階干擾項
- 論證結構分析

### 最深（difficulty=5）
- 跨篇章比較（需同時理解兩篇）
- 辨析高度近似的概念差異
- 要求多維度綜合判斷

## 八、批量寫入腳本規範

### 8.1 檔案：`batch_questions.py`

這是所有手動批量題目的**唯一入口**。未來每次生成題目時，**追加到此檔案的 `QUESTIONS` 列表**，而非建立新檔案。

### 8.2 固定結構

```python
"""
Batch insert MC questions into Supabase.
Usage:
    python batch_questions.py --dry-run   # Preview only
    python batch_questions.py             # Write to DB
"""
from __future__ import annotations
import argparse, json, uuid, sys
from pathlib import Path

# ── Mappings ──
SKILL_TO_TAG = {

    "人物分析": "t-character",
```

### 8.3 單個題目的字典格式

```python
{
    "passage_id": "p01",          # 必填：p01-p12
    "cross_passage": "p02",       # 選填：跨篇章題的第二篇（無則省略此欄）
    "difficulty": "中",            # 必填：最淺/淺/中/深/最深
    "skill": "內容理解",           # 必填：8 種技能之一
    "stem": "根據《論仁、論孝、論君子》，「君子求諸己，小人求諸人」最能體現……",  # 必填
    "options": [                   # 必填：恰好 4 個
        {
            "text": "選項文字",              # 8-15 字
            "is_correct": False,             # 恰好 1 個 True
            "explanation": "此選項錯誤。……"  # 評卷參考級
        },
        # ... 共 4 個
    ],
    "critique_score": 8,           # 自評分（1-10，建議 ≥7）
}
```

### 8.4 `build_rows()` 和 `write_to_supabase()` 函數

這兩個函數已在 `batch_questions.py` 中定義好：
- `build_rows()`：將 QUESTIONS 轉換為 DB 行格式（question row + option rows + tag rows）
- `dry_run()`：以友好格式印出所有題目供檢查
- `write_to_supabase()`：**只寫入新題目**——先從 DB 抓取所有已有 stem，跳過已存在的題目，僅 insert 新題

### 8.5 每次新增題目的流程

1. **查詢現有題庫（僅相關篇章）**：只查詢即將出題的 passage_id 的已有題幹，不載入整個題庫
2. **規劃分佈**：決定新一批題目的 passage × skill × difficulty 分佈，填補空缺
3. **撰寫題目**：按照本指引的所有規範，確保不與已有題目重複
4. **追加到 QUESTIONS 尾部**：在列表末尾追加新題目字典
5. **更新 `PROCESSED_UP_TO`**：將此變量設為追加前的 QUESTIONS 長度（即新批次的起始索引）
6. **執行 dry-run**：`python batch_questions.py --dry-run`（自動只處理新題目）
7. **執行寫入**：`python batch_questions.py`（自動只處理新題目，只查詢相關篇章的已有 stem）
8. **上傳後更新 `PROCESSED_UP_TO`**：設為新的 QUESTIONS 總長度

> **重要效率原則**：
> - `PROCESSED_UP_TO` 變量標記了已入庫題目的數量。腳本默認只處理 `QUESTIONS[PROCESSED_UP_TO:]`，不重新處理歷史題目。
> - `write_to_supabase()` 只查詢當前批次涉及的 passage_id 的已有 stem，不掃描整個題庫。
> - 使用 `--all` 可處理全部題目（僅用於調試），`--from-index N` 可手動指定起始索引。

### 8.6 LLM Agent 高效出題原則

> **核心原則：不做多餘的工作。**

1. **不重新推送已入庫的題目**  
   LLM 應清楚記住哪些題目已經成功寫入 DB。**絕不**把已入庫的題目重新推送再讓腳本去重——這是浪費 API 調用和時間。

2. **不掃描整個題庫做去重**  
   去重應在生成階段完成：LLM 先讀取目標篇章的已有 stem（`sb.table("dsemcq_questions").select("stem").eq("passage_id", "pXX")`），然後生成不重複的新題目。腳本的 stem 去重只是最後防線，不應作為常規工作流程。

3. **只查詢相關篇章**  
   生成 p06 的題目時，只需查詢 p06 的已有 stem，不需要載入 p01–p05 的 911 條題目。

4. **`PROCESSED_UP_TO` 必須及時更新**  
   每次成功上傳後，立即更新此變量。這確保下次運行只處理新增的題目。

5. **批次標註清晰**  
   每批題目用 `# Batch N – pXX × 技能 × 數量` 的註釋標記，方便追蹤和定位。

## 九、品質核查清單

在提交任何題目前，逐項確認：

### 格式
- [ ] 題幹包含篇章名稱（`《》` 或子篇名）
- [ ] 題幹無 p01/p10 等內部 ID
- [ ] 4 個選項長度對稱（誤差 ≤ 30%）
- [ ] 每個選項 8–15 字

## 十一、常見錯誤與教訓（從審計中總結）

### 錯誤 1：大量重複同一句子
六國論（p10）有 165 題，其中大量重複「賂秦根本原因」、「今日割五城」、「第二段論證主旨」等。  
**教訓**：每寫一批新題前，先按 passage 查看所有已有題幹，主動避開已覆蓋的句子。

### 錯誤 2：題幹用「本文」「全文」而不點名篇章
44 題使用了「本文」「全文」「節錄」等模糊指稱，學生在 app 中看到時不知道考哪篇。  
**教訓**：永遠在題幹中用 `《篇章名》` 或引用原文 `「」` 讓學生可辨識。

### 錯誤 3：暴露內部 passage ID
9 題在題幹中出現了 `(p10)`、`p11` 等內部編號。  
**教訓**：寫完每道題後，搜索是否含有 `p\d+` pattern。

### 錯誤 4：同一措辭微調產生偽多樣性
例如同一題「下列哪一項最能概括……」vs「下列哪一項最能準確概括……」，句意相同但各算一題。  
**教訓**：比較新題時，不只看措辭差異，更要看**引用的原文句子 + 考核的具體問題**是否真正不同。

### 錯誤 5：跨篇章題佔比過低
330 題中跨篇章題不足 15 題。目標佔比 10–15%。  
**教訓**：每 20 題中至少安排 2–3 題跨篇章比較題。
